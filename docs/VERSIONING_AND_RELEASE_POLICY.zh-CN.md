# 版本号与发布策略

## 当前基线

- Rust 工作区版本：`0.1.0`
- 桌面端 package 版本：`0.1.0`
- Tauri 应用版本：`0.1.0`
- 当前 `CHANGELOG.md` 中的 `0.1.0` 表示仓库基线，而不是已经完成签名与公开发布的正式版本。

## 首个正式公开版本目标

建议首个正式公开版本使用：

- `v0.2.0`

### 为什么不是 `0.1.0`

- `0.1.0` 已经被当前仓库内部基线占用。
- 当前仓库还没有形成签名安装包、公开发布自动化或广泛公开分发基线。
- 使用 `0.2.0` 可以把“当前内部基线”与“首次正式对外发布”清楚地区分开。

## 建议的发布顺序

1. `0.1.0`
   - 当前内部仓库基线
   - 不代表广泛公开分发
2. `0.2.0-rc.1`、`0.2.0-rc.2` ...
   - 如果正式发布前需要外部试用，可先发布 RC 版本
3. `0.2.0`
   - 首个正式公开版本

## 版本号递增规则

在 `1.0.0` 之前，建议采用以下实用规则：

- `patch`（如 `0.2.1`）
  - bug 修复
  - 打包 / 构建修复
  - 不改变产品范围的文档修正
  - 不带明显用户可见能力变化的测试 / 工具修正
- `minor`（如 `0.3.0`）
  - 明显的用户工作流新增或增强
  - 会影响公开能力面的 CLI parity 提升
  - 外部用户/集成方能感知到的导出或契约扩展
  - 会明显改变使用方式的发布流程改进
- `pre-release`（如 `0.2.0-rc.1`）
  - 下一个 minor 正式版之前的公开验证阶段

## 需要同步更新的文件

切首个正式公开版本时，以下文件应一起更新：

- `Cargo.toml` → `[workspace.package].version`
- `apps/desktop/package.json` → `version`
- `apps/desktop/src-tauri/tauri.conf.json` → `version`
- `CHANGELOG.md`
- `README.md`
- `README.zh-CN.md`
- 发布说明 / Git tag

## `v0.2.0` 的建议发布门槛

在首个正式公开版本打 tag 之前，建议至少满足：

- `pnpm test` 通过
- `pnpm build` 通过
- `pnpm e2e` 通过
- `cargo test --workspace` 通过
- 维护者本机上的 `pnpm qa:full-regression` 保持健康
- README 中的截图与安装说明已更新
- 基础 CI 通过
- 公开仓库治理文件已经到位（`LICENSE`、`CONTRIBUTING`、`CODE_OF_CONDUCT`、`SECURITY`）
- 打包 / 签名状态有明确公开说明，即使签名仍未完全到位

## Tag 与发布说明约定

- Git tag 建议使用 `v0.2.0` 这种格式
- 预发布建议使用 `v0.2.0-rc.1` 这种格式
- 创建 tag 前先更新 `CHANGELOG.md`
- 对外发布说明建议至少包含：
  - 当前能力范围
  - 已知限制
  - 平台支持范围
  - 验证状态

## 针对当前仓库的实际说明

在首个正式公开版本真正发布之前，`0.1.0` 仍应被视为“当前仓库基线”，而不是对外承诺稳定性的正式版本。
